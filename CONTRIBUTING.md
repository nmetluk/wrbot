# CONTRIBUTING — соглашения проекта

Документ обязателен к прочтению и для людей, и для ИИ-агентов.

## Git и ветвление
- **Источник правды — `origin/main` на GitHub.** Любая сессия начинается с `git pull`.
- Малые/инфраструктурные изменения допускаются прямо в `main`.
- Крупные фичи — ветка `feat/TASK-XXXX-кратко`, затем PR в `main`
  (шаблон `.github/pull_request_template.md`). CI должен быть зелёным.
- Никогда не переписывай опубликованную историю (`push --force` в `main` запрещён).

## Conventional Commits
Формат: `<type>(<scope>): <subject>` и ссылка на задачу в теле.

```
feat(charges): добавить пошаговое создание списания

Реализованы шаги 1–7 диалога из ТЗ §3.2.

Task: TASK-0007
```
Типы: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `ci`, `audit`.

## Сообщения и язык
- Документация, задачи, отчёты, коммиты — на русском (термины/код — как принято в индустрии).
- Интерфейс бота — только русский (ТЗ §4).

## Качество кода (для исполнителя)
- Python 3, форматирование `ruff format`, линт `ruff check`, типизация `mypy` (где применимо).
- Тесты `pytest`. Новая логика — с тестами. CI запускает их на каждый push.
- Чистый, документированный код (ТЗ §5): docstring у публичных функций, понятные имена.
- **Pre-commit hooks** (TASK-0009): после клонирования выполни `pre-commit install`.
  Это автоматически запускает `ruff format` и `ruff check --fix` перед каждым коммитом,
  предотвращая дрейф форматирования.
- **Инструменты через uv (TASK-0020):** для локального прогона гейта (как в CI) используй
  `uv run ruff ...`, `uv run mypy src`, `uv run pytest` и т.д. (после `uv sync --frozen --extra dev`).
  Это гарантирует те же версии, что в `uv.lock` (ruff==0.15.15, mypy==2.1.0 и др.).

## Безопасность
- Секреты — только в `.env` (см. `.env.example`). Токен бота, PAT, строки подключения к БД — не коммитятся.
- Пользовательские данные хранятся по `tg_id`, изолированно (ТЗ §1.3).
- Любая находка по безопасности фиксируется в отчёте аудита (`handoff/reports/`).

## Определение «Готово» (Definition of Done)
Задача переходит в `done/`, только если:
1. Реализованы все пункты задачи (acceptance criteria).
2. CI зелёный (lint, тесты, валидация handoff/state).
3. Написан отчёт `handoff/reports/TASK-XXXX-report.md`.
4. Обновлены `state/project.json`, `state/CHANGELOG.md`, создан лог сессии.
5. Изменения закоммичены и **запушены** в `origin/main`.
