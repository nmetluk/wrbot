# Протокол handoff

Контракт, по которому архитектор и исполнитель обмениваются задачами через git.
**Источник правды — `origin/main`.** Любое изменение конвейера — это коммит+push.

## Машина состояний задачи
```
inbox/ ──/take-task──► in-progress/ ──успех──► done/   (+ reports/TASK-XXXX-report.md)
   ▲                        │
   └──архитектор отвечает── blocked/  ◄──нужно решение/уточнение──┘
```

## Файл задачи
- Путь: `handoff/inbox/TASK-XXXX-краткое-латиницей.md`.
- Номер `XXXX` — сквозной, монотонный, не переиспользуется (см. `state/backlog.json`).
- Структура — по шаблону `docs/templates/task.template.md`. Обязательно:
  цель, контекст/ссылки (ТЗ §, FR/NFR ID), **критерии приёмки**, ограничения,
  ожидаемые артефакты, оценка майлстоуна.

## Отчёт исполнителя
- Путь: `handoff/reports/TASK-XXXX-report.md` (по `docs/templates/report.template.md`).
- Обязательно: что сделано, как проверено (тесты/CI), отклонения от задачи,
  затронутые файлы, ссылки на коммиты, открытые вопросы.

## Переходы (как именно)
| Действие | Кто | Команда/скрипт | Эффект |
|----------|-----|----------------|--------|
| Создать задачу | архитектор | `/new-task` → `scripts/new_task.py` | файл в `inbox/`, запись в `backlog.json` |
| Взять задачу | исполнитель | `/take-task` → `scripts/take_task.py` | `git mv` верхней задачи `inbox/`→`in-progress/` |
| Завершить | исполнитель | `/end-session` → `scripts/complete_task.py` | `git mv` →`done/`, отчёт в `reports/`, лог сессии, push |
| Заблокировать | исполнитель | вручную/скриптом | `git mv` →`blocked/` + причина в файле/отчёте |
| Разблокировать | архитектор | правка + `git mv` →`inbox/` | задача снова доступна |

После **любого** перехода: `python scripts/update_state.py` пересобирает
`state/backlog.json`; CI (`validate.yml`) проверяет соответствие каталогов и индекса.

## Выбор «верхней» задачи
`/take-task` берёт задачу из `inbox/` с **наименьшим** номером `TASK-XXXX`, если у
неё нет неудовлетворённых зависимостей (поле `Зависит от` в файле задачи). Это
даёт детерминированный порядок и сохраняет преемственность контекста.

## Правила целостности
- Одна задача в `in-progress/` за раз на одного исполнителя (иначе — ветки и дисциплина PR).
- Перемещения — только `git mv` (сохраняем историю).
- Нельзя класть в `done/` задачу без отчёта и зелёного CI (Definition of Done в `CONTRIBUTING.md`).
