# Гайд исполнителя (Claude Code)

Исполнитель реализует задачи. **ТЗ и архитектурные решения не меняет** без задачи —
при конфликте/неясности переводит задачу в `blocked/`.

## Триггер пользователя
Пользователь говорит: **«возьми задачу исполнителя»**. Тогда исполнитель:
1. Ритуал начала сессии (`git pull`, `AGENTS.md`, `state/`, последний лог).
2. `/take-task` — берёт верхнюю задачу из `inbox/` (мин. номер без незакрытых зависимостей),
   переносит её в `in-progress/`.
3. Выполняет строго по критериям приёмки, обновляя `docs/` и `src/` по необходимости.
4. Гоняет локально lint/тесты (как в CI).
5. `/end-session` — пишет отчёт в `reports/`, переносит задачу в `done/`, создаёт лог
   сессии, обновляет `state/`, коммитит и **пушит** (PAT уже на машине).

## Правила
- Один фокус. Нашёл смежную работу — заведи новую задачу в `inbox/`, не раздувай текущую.
- Любое значимое решение — ADR (`docs/architecture/decisions/`), затем продолжай.
- Сомнение в требовании → `blocked/` + конкретный вопрос архитектору, не «угадывай».
- Секреты только в `.env`. Токен бота/PAT/строку БД в репозиторий не коммить.
- Conventional Commits со ссылкой `Task: TASK-XXXX`.

## Definition of Done
Перед переносом в `done/` убедись (см. `CONTRIBUTING.md`):
1. Все критерии приёмки выполнены.
2. CI зелёный (lint, типы, тесты, валидация handoff/state).
3. Отчёт написан.
4. `state/` и `CHANGELOG.md` обновлены, лог сессии создан.
5. Запушено в `origin/main`.

### Обязательный прогон CI начисто (НЕ заявляй «зелёно» без него)
Прогони **ровно те же команды, что и `.github/workflows/ci.yml`**, в чистом окружении
**без `BOT_TOKEN`**, и приведи их фактический вывод в отчёте:

```
ruff format --check src tests
ruff check src tests
mypy src
pytest
alembic upgrade head      # без BOT_TOKEN
python scripts/validate.py
```

Урок M2 (TASK-0005): отчёт заявил «всё зелёное», но `mypy` не прогонялся, а `ruff check
src tests` был красным (64 ошибки mypy + E501). **«Зелёно» = каждая из этих команд
вернула 0 ошибок при реальном запуске.** `pytest` сам по себе НЕ равен зелёному CI.

### Тестируй то, что unit-тесты не ловят
Юнит-тесты, вызывающие хэндлеры напрямую, **не проверяют диспетчеризацию роутера** и
порядок фильтров. Для callback-роутинга и пошаговых сценариев добавляй тест через сам
роутер/диспетчер (урок TASK-0008: широкий `startswith` перехватывал спец-callback'и).

## Если что-то пошло не так
- CI красный — чини до зелёного, задачу в `done/` не переноси.
- Не хватает решения архитектора — `blocked/`, опиши блокер, заверши сессию с логом.
