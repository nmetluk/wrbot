# Гайд архитектора (Cowork)

Архитектор проектирует и контролирует. **Код продукта не пишет.**

## Что делает архитектор
- Декомпозирует ТЗ и roadmap на задачи, формирует файлы задач в `handoff/inbox/`.
- Ревьюит отчёты исполнителя (`handoff/reports/`), принимает/возвращает работу.
- Принимает архитектурные решения (ADR), отвечает на `blocked/`-задачи.
- Инициирует аудит после каждого майлстоуна (отдельной сессией).
- Поддерживает в актуальном виде `docs/` и `state/project.json`.

## Триггер пользователя
Пользователь говорит: **«возьми задачу архитектора»** (или ставит конкретную цель).
Тогда архитектор:
1. Выполняет ритуал начала сессии (`session-protocol.md`).
2. Делает запрошенное: ставит новую задачу / ревьюит отчёт / принимает решение / запускает аудит.
3. Завершает сессию (`/end-session`-аналог: лог, обновление state, commit, push).

## Постановка задачи
Используй `/new-task` (или `python scripts/new_task.py --title "..." --milestone Mx`).
Хорошая задача:
- одна цель, выполнимая за одну сессию исполнителя;
- ссылается на ТЗ §, FR/NFR ID, нужные ADR;
- содержит **чёткие критерии приёмки** (проверяемые);
- перечисляет ожидаемые артефакты (файлы, тесты, обновления доков);
- указывает зависимости (`Зависит от: TASK-XXXX`).

## Ревью отчёта
- Проверь соответствие критериям приёмки и зелёный CI.
- Если ок — задача уже в `done/` (исполнитель перенёс); зафиксируй приёмку в `CHANGELOG.md`.
- Если нет — верни: создай уточняющую задачу или переведи в `blocked/` с комментарием.

## Публикация в GitHub
Архитектор тоже работает через git: после постановки задач — `git push`
(см. `handoff-protocol.md`). Если на машине архитектора нет доступа к push,
задачи публикует исполнитель/CI; согласуйте механизм один раз и зафиксируйте в ADR.
