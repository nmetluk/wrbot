# Протокол сессии

Каждый шаг проекта = отдельная сессия. Сессия эфемерна; всё ценное оседает в репозитории.

## Начало сессии (обязательный ритуал)
1. `git pull` — синхронизация с источником правды.
2. Прочитать `AGENTS.md`.
3. Прочитать `state/project.json` (фаза, активная задача, next_step).
4. Прочитать последний файл в `sessions/`.
5. Открыть гайд своей роли (`architect-guide.md` / `executor-guide.md`).

## Во время сессии
- Держать фокус на одной задаче. Расширение scope — новой задачей в `inbox/`.
- Принятые значимые решения сразу оформлять как ADR.
- Коммитить осмысленными порциями (Conventional Commits).

## Конец сессии (обязательно)
1. Создать лог `sessions/SESSION-YYYY-MM-DD-NN.md` (шаблон `docs/templates/session-log.template.md`).
   **ID уникален.** Если на дату уже есть лог с этим `NN` — увеличь `NN` (01 → 02 → …).
   **Никогда не перезаписывай существующий лог сессии** (в нескольких сессиях за день
   участвуют разные роли/агенты; затирание уничтожает их контекст).
2. Обновить `state/project.json` (`updated_at`, `current_phase`, `active_task`,
   `last_session`, `next_step`).
3. Запустить `python scripts/update_state.py` (пересбор `backlog.json`).
4. Дополнить `state/CHANGELOG.md`.
5. Закоммитить и **запушить** в `origin/main`.

Команда `/end-session` выполняет шаги 1–5 полуавтоматически.

## Почему так
Следующая сессия (возможно, другой агент, другая ОС, другой человек) должна
восстановить контекст **только из репозитория**. Если после сессии этого
недостаточно — протокол нарушен. См. `context-preservation.md`.
