# Протокол аудита

После завершения **каждого крупного раздела/майлстоуна** (см. `docs/roadmap.md`)
проводится комплексный аудит — **в отдельной сессии**, чтобы взгляд был
независимым и не «замыленным» реализацией.

## Когда
- По завершении майлстоуна M1…M5.
- Перед релизом/деплоем.
- По требованию архитектора (например, после крупного рефакторинга).

## Направления аудита
1. **Безопасность.** Секреты вне репозитория и истории; изоляция данных по `tg_id`
   (FR-13); валидация пользовательского ввода; обработка ошибок Telegram API;
   зависимости без известных уязвимостей.
2. **Архитектура.** Соблюдение слоёв и границ; соответствие ADR; нет «протечек» БД в хэндлеры;
   расширяемость под §6 ТЗ (NFR-4).
3. **Качество кода.** Линт/типизация/покрытие тестами; читаемость; docstring; дубли.
4. **Соответствие требованиям.** Трассировка FR/NFR ↔ реализация ↔ тесты; пробелы.
5. **Надёжность 24/7.** Поведение планировщика при рестарте; точность ±1 мин (NFR-2);
   идемпотентность напоминаний; логирование.
6. **Процесс/контекст.** Целостность `handoff/` и `state/`; полнота логов сессий; ADR на месте.

## Как
1. Отдельная сессия с ролью «аудитор» (может вести архитектор или выделенный агент).
2. Прогон автоматики: `python scripts/validate.py`, CI, `ruff`, `mypy`, `pytest`,
   аудит зависимостей (`pip-audit`/`safety`).
3. Заполнить отчёт по шаблону `docs/templates/audit-report.template.md`,
   сохранить в `handoff/reports/AUDIT-Mx-YYYY-MM-DD.md`.
4. Находки оформить как задачи в `handoff/inbox/` (с приоритетом), критичные по
   безопасности — немедленно.
5. Зафиксировать итог в `state/CHANGELOG.md` и лог сессии.

## Критерий прохождения
Майлстоун считается принятым, когда аудит без критичных находок (или они закрыты
отдельными задачами и перепроверены).
