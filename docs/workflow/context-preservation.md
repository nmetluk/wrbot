# Сохранение контекста между сессиями

Главный риск модели «каждый шаг — отдельная сессия» — потеря контекста. Здесь —
как мы с этим боремся. Цель: **новый агент или человек без доступа к чату
полностью восстанавливает картину за ≤ 5 минут чтения репозитория.**

## Слои памяти проекта
| Слой | Где | Для кого | Горизонт |
|------|-----|----------|----------|
| Точка входа | `AGENTS.md`, `.claude/CLAUDE.md` | агент в начале сессии | постоянный |
| Снимок состояния | `state/project.json` | агент/человек | «сейчас» |
| Индекс задач | `state/backlog.json` | агент/CI | «сейчас» |
| Нарратив | `sessions/SESSION-*.md` | человек/агент | история |
| Хроника | `state/CHANGELOG.md` | человек | история |
| Решения | `docs/architecture/decisions/` | агент/человек | постоянный |
| Задачи и отчёты | `handoff/` | обе роли | по задаче |
| Код и история | `src/`, `git log` | исполнитель | постоянный |

## Принципы
1. **Дублируй для двух читателей.** Нарратив (Markdown) — человеку; структура
   (JSON) — машине. Один факт живёт в обоих представлениях.
2. **Пиши для незнакомца.** Никаких «как обсуждали» — только самодостаточные записи.
3. **Связывай.** Задачи ссылаются на FR/NFR и ADR; отчёты — на коммиты; ADR — на сессии.
4. **Состояние = производная от репозитория.** `backlog.json` генерируется из `handoff/`,
   не ведётся вручную; CI ловит расхождения.
5. **Ничего важного только в голове сессии.** Перед `end-session` спроси себя:
   «достаточно ли в репозитории, чтобы продолжить с нуля?»

## Чек-лист «контекст сохранён»
- [ ] `state/project.json` отражает реальность (фаза, активная задача, next_step).
- [ ] Есть лог текущей сессии в `sessions/`.
- [ ] Значимые решения оформлены как ADR.
- [ ] Отчёт по задаче в `handoff/reports/` (если задача затронута).
- [ ] Всё запушено в `origin/main`.
