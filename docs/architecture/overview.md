# Обзор системы

## Контекст
Один процесс-бот, общающийся с Telegram Bot API (long polling на старте; webhook —
опционально позже), с локальной БД и встроенным планировщиком напоминаний.

```
        ┌─────────────┐   updates    ┌──────────────────────────────────────┐
        │  Telegram    │◄────────────►│              wrbot (1 процесс)        │
        │  Bot API     │   sendMessage│                                       │
        └─────────────┘              │  ┌────────────┐   ┌─────────────────┐ │
                                     │  │ Handlers/   │   │  Scheduler       │ │
   Пользователь ──/start, кнопки──►  │  │ FSM-диалоги │   │  (APScheduler)   │ │
                                     │  └─────┬───────┘   └────────┬────────┘ │
                                     │        │  service layer     │          │
                                     │        ▼                    ▼          │
                                     │     ┌───────────────────────────┐      │
                                     │     │  Repositories (data access)│      │
                                     │     └─────────────┬─────────────┘      │
                                     └───────────────────┼────────────────────┘
                                                         ▼
                                                  ┌─────────────┐
                                                  │  БД          │
                                                  │ SQLite→PG    │
                                                  └─────────────┘
```

## Слои (clean-ish layering)
1. **Presentation / Handlers** — хэндлеры Aiogram, inline-клавиатуры, FSM пошаговых
   диалогов (создание/редактирование списания, CRUD кошельков/категорий, настройки).
2. **Service / Domain** — бизнес-логика: расчёт следующей даты по периодичности,
   определение, кому и когда слать напоминание, обработка «Оплачено/Отложить».
3. **Data access / Repositories** — изоляция БД; CRUD по сущностям, всё в разрезе `tg_id`.
4. **Scheduler** — APScheduler: периодическая проверка/постановка напоминаний с
   точностью ±1 минута (NFR-2); устойчивость к перезапуску (jobstore/пересборка из БД).
5. **Infrastructure** — конфиг из окружения (токен, строка БД, TZ), логирование,
   точка входа, миграции схемы.

## Ключевые решения, требующие проработки (ADR)
- Способ хранения и срабатывания напоминаний (cron-проверка vs. индивидуальные джобы).
- Стратегия часовых поясов и трактовка `notify_time`.
- Слой доступа к данным (сырой SQL vs. SQLAlchemy/aiosqlite) с учётом миграции SQLite→PostgreSQL.
- Семантика кнопки «Отложить».

См. `decisions/`. До принятия решения реализация по этим пунктам не начинается.

## Нефункциональные опоры
- **24/7 (NFR-1):** процесс под супервизором (systemd/докер-рестарт), идемпотентная
  отправка, отсутствие потери джоб при рестарте.
- **Точность ±1 мин (NFR-2):** планировщик с минутной гранулярностью, защита от дублей.
- **Расширяемость (NFR-4):** чёткие границы слоёв, репозитории как абстракция БД.
