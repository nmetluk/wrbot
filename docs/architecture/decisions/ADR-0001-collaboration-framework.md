# ADR-0001: Фреймворк совместной работы (handoff через git)

- **Статус:** Accepted
- **Дата:** 2026-05-31
- **Контекст сессии:** SESSION-2026-05-31-01
- **Связанные задачи:** —

## Контекст
Над проектом работают две роли в разных средах: **архитектор** в Cowork и
**исполнитель** в Claude Code на отдельной машине (с PAT). Каждый шаг выполняется
в **отдельной сессии**, контекст между сессиями не сохраняется автоматически.
Исполнители разных этапов могут быть на разных ОС. Нужна схема, при которой:
- контекст максимально передаётся между сессиями;
- логи/сессии читаемы и человеком, и другим ИИ-агентом;
- нет ручных рутинных команд;
- есть единый источник правды.

## Рассмотренные варианты
1. **Файлы задач в git + GitHub как источник правды (выбрано).** Задачи и отчёты —
   Markdown-файлы в каталогах-состояниях `handoff/`, состояние дублируется в `state/*.json`.
2. **GitHub Issues как очередь задач.** Нативный трекинг, но требует API/PAT у обеих
   ролей и выносит контекст из репозитория (хуже для оффлайн-восстановления ИИ-агентом).
3. **Внешний трекер (Jira/Trello/Notion).** Лишняя зависимость, не «источник правды = git».

## Решение
Выбран вариант 1. Всё — в репозитории:
- `handoff/{inbox,in-progress,done,blocked,reports}` — конвейер задач (машина состояний).
- `sessions/` — append-only логи сессий (нарратив для человека и ИИ).
- `state/{project.json,backlog.json,CHANGELOG.md}` — машиночитаемый снимок состояния.
- `AGENTS.md` + `.claude/CLAUDE.md` — точка входа в контекст для агента.
- Синхронизация ролей — только `git pull`/`git push`; **GitHub — единственный источник правды**.
- Автоматизация — кроссплатформенные Python-скрипты (`scripts/`) и слэш-команды Claude Code.

## Последствия
- **+** Полный контекст восстанавливается без доступа к чату; всё под версионным контролем.
- **+** Кроссплатформенно, без внешних зависимостей, дёшево.
- **+** CI может валидировать целостность конвейера и состояния.
- **−** Нет нативной доски задач (компенсируется `state/backlog.json` и, при желании, артефактом).
- **−** Параллельная работа над одной задачей требует дисциплины веток (см. CONTRIBUTING.md).
