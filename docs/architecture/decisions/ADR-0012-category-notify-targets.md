# ADR-0012: Дублирование напоминаний в каналы/группы по категориям

- **Статус:** Accepted
- **Дата:** 2026-06-03
- **Контекст сессии:** SESSION-2026-06-03-16
- **Связанные задачи:** TASK-0043 (схема + UI), TASK-0044 (отправка + обработка ошибок)

## Контекст
Отзыв пользователей: хочется, чтобы напоминания о списаниях дублировались в выбранные
Telegram-каналы/группы — на уровне категории (все списания категории шлются в её чаты).
Если бот не имеет прав постить в указанный чат — уведомить владельца о неудаче.
Ограничения проекта: только `aiogram.Bot` (не прямой HTTP к Telegram), все тексты — через
локали, доступ к БД — только через репозитории, async-only, не логировать чувствительное.

## Рассмотренные варианты
1. **Список chat_id в категории (JSON-колонка `Category.notify_chat_ids`).** Плюсы: просто,
   без новой таблицы, достаточно для «несколько чатов на категорию». Минусы: нет метаданных
   на цель (название/статус) — компенсируется хранением последней ошибки отдельно/в логах.
2. **Отдельная таблица `category_notify_targets` (FK на категорию, по строке на чат).** Плюсы:
   расширяемость (статус прав, заголовок чата, last_error). Минусы: миграция + репозиторий +
   больше кода; для текущего объёма избыточно.

## Решение
Старт с **варианта 1**: nullable JSON-колонка `Category.notify_chat_ids` (список int chat_id,
по умолчанию пусто/NULL). Управление в настройках категории: добавить/удалить chat_id
с валидацией (целое, формат канала/группы; @username не поддерживаем в v1 — только числовой id).
Отправка (TASK-0044): при рассылке напоминания по списанию, кроме личного чата владельца,
бот постит копию в каждый `notify_chat_id` категории этого списания через `aiogram.Bot`.
**Изоляция ошибок:** сбой отправки в один чат не ломает остальные и не ломает личное напоминание.
При `TelegramForbiddenError`/`TelegramBadRequest` (нет прав/бот не в чате) — однократно уведомить
владельца в личку: «не удалось отправить в <chat_id>, проверьте права бота» (текст — через локаль).
Эскалация на v2: при необходимости — миграция на вариант 2 (таблица со статусом прав).

## Последствия
- Положительные: минимальная схема (одна колонка), переиспользуется существующий путь отправки
  напоминаний; владелец узнаёт о проблемах с правами.
- Отрицательные / компромиссы: нет постоянного хранения статуса прав/заголовка чата; проверка
  прав — реактивная (по факту ошибки отправки), а не заранее; только числовые chat_id в v1.
- Безопасность: chat_id — не секрет, но не логировать содержимое напоминаний; уведомления о сбое
  идут владельцу, не в админ-канал. Дубль-отправка не должна влиять на идемпотентность
  `SentReminder` (учёт ведём по владельцу/charge, копии в чаты — best-effort, см. TASK-0044).
- Разблокирует: TASK-0043 → TASK-0044.
