# ADR-0010: Аудит-лог действий

- **Статус:** Accepted
- **Дата:** 2026-06-03
- **Контекст сессии:** SESSION-2026-06-03-03
- **Связанные задачи:** TASK-0032 (M6)

## Контекст
Нужен журнал действий пользователей и админа — для расследований и подотчётности, без
нарушения приватности (ТЗ: не логировать личные данные/суммы/контакты).

## Решение
- Новая таблица `audit_log` (Alembic-миграция):
  `id`, `actor_id` (tg_id или admin), `actor_role` (`user`/`admin`/`system`), `action`
  (enum-строка, напр. `charge.create`, `charge.paid`, `charge.delete`, `wallet.create`,
  `category.delete`, `settings.notify_time`, `tz.change`, `admin.backup`, …),
  `entity_type` (`charge`/`wallet`/`category`/`user`/null), `entity_id` (int|null),
  `created_at` (UTC). **Без чувствительных данных** (сумм, имён, текстов, контактов).
- Запись — через `AuditLogRepository.record(...)`, вызывается из сервисов/хэндлеров после
  успешной мутации (точечно, не «голым» middleware, чтобы фиксировать семантику действия).
- **Дубль критичных событий в админ-канал** (ADR-0008): ошибки уровня бота, админ-действия,
  потенциально массовые операции. Обычные пользовательские действия — только в таблицу.
- Изоляция по tg_id сохраняется; чтение лога — административная функция (вне scope v1.1 UI,
  достаточно таблицы + дублей в канал).

## Последствия
- **+** Подотчётность и базовая форензика; данные для дневного дашборда (активные пользователи,
  ошибки за день — ADR-0008/TASK-0034).
- **+** Приватность соблюдена (нет сумм/имён в логе).
- **−** Дополнительные записи в БД на каждое действие (объём малый; при росте — ротация/архивация отдельной задачей).
- Требует FK `audit_log.actor_id`? Нет — actor может быть admin/system; храним как int без FK,
  чтобы не терять записи при удалении пользователя.
