# ADR-0006: Жизненный цикл списания и формат суммы

- **Статус:** Accepted
- **Дата:** 2026-06-01
- **Контекст сессии:** SESSION-2026-06-01-01
- **Связанные задачи:** TASK-0001 (схема), M3 (логика «оплачено»)

## Контекст
FR-7: «Отметить как оплачено» сдвигает `next_date` на период, а одноразовое
«закрывается». ТЗ не уточняет, что значит «закрывается» (удалить vs пометить), и
не фиксирует формат суммы. От этого зависит схема и возможность статистики (§6 ТЗ).

## Решение
**Жизненный цикл (мягкое закрытие):**
- У `charges` есть статус: поле `status` (`active` / `done`) и `paid_at` (UTC).
- «Оплачено» по **периодическому** списанию: `next_date += период` (с клампом конца
  месяца — см. ниже), статус остаётся `active`, событие платежа фиксируется (для будущей истории).
- «Оплачено» по **одноразовому**: `status = done`, `paid_at = now`. Запись **не удаляется**,
  скрывается из «Мои списания», но доступна для статистики §6.
- Кламп конца месяца: при `monthly`/`quarterly`/`yearly`, если целевого числа нет в месяце
  (например, 31-е в феврале), дата клампится к последнему дню месяца.

**Формат суммы:**
- Хранить как **Decimal/NUMERIC** (две знака после запятой), не float (исключаем
  ошибки округления денег).
- v1 — одна валюта (рубль), отображение `₽`; отдельной колонки валюты пока нет
  (несколько валют — §6, вторая итерация). Заложить так, чтобы добавление валюты
  не ломало схему (миграция-расширение).

## Последствия
- **+** История платежей сохраняется → статистика §6 реализуема без рефакторинга.
- **+** Корректная денежная арифметика.
- **−** «Мои списания» и репозитории обязаны фильтровать по `status = active`.
- **−** Полноценная история отдельных платежей (журнал) может потребовать отдельной
  таблицы во второй итерации; в v1 достаточно `status` + `paid_at` на самом списании.
