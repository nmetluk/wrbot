# ADR-0003: Слой доступа к данным и миграции

- **Статус:** Accepted
- **Дата:** 2026-06-01
- **Контекст сессии:** SESSION-2026-06-01-01
- **Связанные задачи:** TASK-0001

## Контекст
ТЗ §4 требует путь **SQLite (старт) → PostgreSQL**, NFR-4 — лёгкую расширяемость.
ADR-0002 оставил выбор слоя данных открытым. Нужно зафиксировать его до TASK-0001,
который закладывает схему БД.

## Рассмотренные варианты
1. **Сырой `aiosqlite` + ручные SQL-миграции** — минимум зависимостей, но
   SQLite→PostgreSQL и эволюция схемы делаются руками; высок риск рассинхрона
   типов (SQLite слабо типизирован) — ровно тот класс багов, против которого
   борется проект.
2. **SQLAlchemy 2.0 (async) + Alembic** — единый ORM-слой над обоими движками,
   автогенерируемые версионируемые миграции, тот же стек, что в соседнем проекте
   `whois-watcher` (проверен в бою). Чуть больше сетапа.

## Решение
**SQLAlchemy 2.0 async + Alembic + asyncpg (для PostgreSQL) / aiosqlite (для SQLite).**
Доступ к данным — только через репозитории (`src/wrbot/repositories/`), как
оговорено в `components.md`. Схема живёт в моделях SQLAlchemy; изменения схемы —
только через миграции Alembic, без `CREATE TABLE` в коде.

## Последствия
- **+** Чистый, проверяемый путь SQLite→PostgreSQL; миграции из коробки; типобезопасность.
- **+** Совпадает со стеком `whois-watcher` — переиспользуемый опыт и паттерны.
- **−** Больше первичного сетапа (alembic.ini, env, базовые модели) в TASK-0001.
- Разблокирует: TASK-0001 (скелет БД), M2/M3 (репозитории сущностей).
- JSON-поля (`global_days`, `individual_days`) хранить как JSON-тип, совместимый
  с обоими движками (SQLite TEXT/JSON ↔ PostgreSQL JSONB).
