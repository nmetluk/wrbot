# ADR-0009: Стратегия резервного копирования БД

- **Статус:** Accepted
- **Дата:** 2026-06-03
- **Контекст сессии:** SESSION-2026-06-03-03
- **Связанные задачи:** TASK-0033 (M6)

## Контекст
Админ справедливо отметил отсутствие бэкапов. Нужны автоматические ежечасные бэкапы с
ротацией и подтверждением в админ-канал. Прод может быть SQLite или PostgreSQL (ADR-0003).

## Решение
- **Ежечасно** (APScheduler) — резервная копия БД в каталог `./backups` (том в Docker).
- **Метод — по типу БД (детект из `DATABASE_URL`):**
  - SQLite → безопасная копия через `VACUUM INTO 'backups/wrbot-YYYYMMDD-HHMM.sqlite3'`
    (консистентный снимок без остановки бота; не просто `cp`).
  - PostgreSQL → `pg_dump` в файл `backups/wrbot-YYYYMMDD-HHMM.sql.gz` (нужен `postgresql-client`
    в образе — дополнить Dockerfile).
- **Ротация:** хранить 36 последних файлов, старшие удалять.
- **Хранение:** только на сервере (том `./backups`); в админ-канал отправляется **статус**
  («✅ Бэкап ОК: <файл>, <размер>» / «❌ Ошибка бэкапа: …»), сами файлы в канал не выгружаются
  (решение владельца). Офсайт (S3) — возможное расширение позже.
- Бэкап-сообщение совмещается с ежечасным heartbeat и краткой сводкой (TASK-0033).

## Последствия
- **+** Восстановление из локальных снимков; консистентность (VACUUM INTO/pg_dump), а не сырой `cp`.
- **+** Переносимо между SQLite и PostgreSQL.
- **−** Нет офсайт-копии (при потере сервера бэкапы теряются) — осознанный компромисс v1.1; S3 — отдельной задачей.
- **−** Для Postgres-пути нужен `postgresql-client` в Docker-образе.
- Требует тома `./backups:/app/backups` в `docker-compose.yml` и каталога с правами non-root пользователя.
