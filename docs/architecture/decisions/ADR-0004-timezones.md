# ADR-0004: Часовые пояса и время уведомлений

- **Статус:** Accepted
- **Дата:** 2026-06-01
- **Контекст сессии:** SESSION-2026-06-01-01
- **Связанные задачи:** TASK-0001 (поле схемы), M4 (использование при отправке)

## Контекст
ТЗ §3.3 задаёт `notify_time` (по умолчанию 10:00), но **не указывает часовой пояс**.
Аудитория русскоязычная — это до 11 часовых поясов. Прислать напоминание об оплате
в неверное время — провал основной функции (NFR-2 «точность ±1 минута» бессмысленна
без корректного пояса).

## Рассмотренные варианты
1. **Фиксированный МСК для всех** — просто, но неверно для пользователей вне МСК.
2. **Поле `tz` у пользователя, дефолт `Europe/Moscow`, UI смены позже** — корректная
   модель данных сразу, минимальный UI на старте.
3. **Полный per-user TZ с экраном выбора уже в v1** — максимально правильно, но
   +работа в UI на первом этапе.

## Решение
**Вариант 2.** В схему `users` добавляется колонка `tz` (IANA-имя, например
`Europe/Moscow`) со значением по умолчанию `Europe/Moscow`. `notify_time` хранится
как локальное время в этом поясе. Расчёт момента отправки = `notify_time` в `tz`
пользователя, приведённое к UTC планировщиком. Экран смены пояса в настройках —
отдельной задачей (может попасть в M2/M4), но **поле схемы заводится уже в TASK-0001**,
чтобы не делать миграцию-ретрофит.

## Последствия
- **+** Корректные напоминания вне МСК; дешёвое поле сейчас вместо дорогой миграции потом.
- **+** Дефолт сохраняет «максимальную простоту» (§1.3) для типичного пользователя.
- **−** Логика отправки (M4) обязана работать в UTC и конвертировать по `tz`; тесты на
  переход через полночь/смену пояса обязательны.
- Все внутренние временные метки (`created_at`, `paid_at`, отметки отправки) — в **UTC**.
