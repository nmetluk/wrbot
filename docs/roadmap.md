# Дорожная карта

Майлстоуны от инфраструктуры до работающего 24/7 бота. **После каждого M(1..5) —
обязательная сессия-аудита** (`docs/workflow/audit-protocol.md`). Задачи каждого
майлстоуна формирует архитектор в `handoff/inbox/`.

| M | Раздел | Содержание | Выход | Аудит |
|---|--------|------------|-------|-------|
| **M0** | Инфраструктура | Каркас репозитория, документация, протоколы, handoff, state, CI, шаблоны | Этот репозиторий | — (ревью) |
| **M1** | Скелет проекта | Структура `src/wrbot`, зависимости, конфиг из окружения, логирование, схема БД, точка входа, «бот отвечает на /start заглушкой» | Запускаемый каркас бота | Безопасность секретов, базовая архитектура |
| **M2** | Справочники | CRUD кошельков и категорий (хэндлеры, репозитории, тесты) | FR-8, FR-9 | Архитектура слоёв, изоляция по tg_id |
| **M3** | Списания | Пошаговое создание (FR-3), «Мои списания» (FR-5), действия и «оплачено» со сдвигом даты (FR-6, FR-7), редактирование/удаление | FR-3..FR-7 | Доменная логика дат/периодов |
| **M4** | Уведомления | APScheduler, глобальные/индивидуальные дни (FR-10, FR-11), отправка с кнопками (FR-12), «отложить» | FR-10..FR-12, NFR-2 | Надёжность 24/7, точность, идемпотентность |
| **M5** | Стабилизация и деплой | Обработка ошибок, устойчивость к рестарту, развёртывание 24/7, README по запуску, нагрузочные/edge-тесты | NFR-1, NFR-3 | Полный комплексный аудит перед релизом |
| **M6** | Наблюдаемость и админ-функции (после v0.1.0) | Админ-канал, ежечасный бэкап БД (ротация 36) + статус/heartbeat + краткая сводка, аудит-лог действий, ежедневный дашборд 21:00 с графиками | ADR-0008…0010 | Безопасность (секреты в канал/лог), надёжность джоб, приватность аудита |
| **M7** | Доработки по отзывам (после v0.2.0) | Фикс отображения (реальные кошелёк/категория, дата ДД.ММ.ГГГГ, период); период→дата + валидация окна; группировка «Мои списания» по категориям; эмодзи-иконки кошельков; дубль напоминаний в каналы/группы по категориям; переработка UX редактирования | ADR-0011…0012 (TASK-0039…0045) | UX/корректность, миграции на Postgres, изоляция ошибок отправки |

## Принципы декомпозиции
- Один майлстоун = несколько задач `TASK-XXXX`, каждая выполнима за одну сессию.
- Зависимости между задачами фиксируются в файлах задач (`Зависит от:`).
- Не начинать M(n+1), пока не закрыт аудит M(n) (критичные находки устранены).
