# TASK-0032: Аудит-лог действий

## Цель
Журналировать действия пользователей и админа в БД (без чувствительных данных), дублируя
критичные события в админ-канал (ADR-0010).

## Контекст
- ADR-0010 (схема, что логируем/не логируем, дубль критичных), ADR-0003 (миграции), ADR-0008 (канал).
- Готово: репозитории/сервисы сущностей (M2–M4), `AdminNotifier` (TASK-0031).

## Критерии приёмки (проверяемые)
- [ ] **Миграция Alembic**: таблица `audit_log` (`id`, `actor_id` int, `actor_role`
      `user`/`admin`/`system`, `action` str, `entity_type` str|null, `entity_id` int|null,
      `created_at` UTC). Индекс по `created_at` и `actor_id`. `autogenerate` даёт пустой diff после.
- [ ] `AuditLogRepository.record(...)` (+ типы действий как константы/enum-строки).
- [ ] Логирование ключевых действий после успешной мутации: charge create/edit/delete/paid/snooze,
      wallet create/rename/delete, category create/rename/delete, settings (notify_time/days/tz).
      **Без сумм/имён/текстов** (только тип+id сущности).
- [ ] Критичные события (ошибки бота из глобального error-handler, админ-действия) **дублируются
      в админ-канал** через `AdminNotifier`.
- [ ] Тесты: запись создаётся при действии (через сервис/хэндлер), отсутствие чувствительных полей,
      дубль критичного в канал (mock), миграция применяется/откатывается.
- [ ] Весь CI начисто зелёный.

## Ожидаемые артефакты
- Код: миграция `alembic/versions/*`, `db/models.py` (модель AuditLog), `repositories/audit_log.py`,
  точки вызова в сервисах/хэндлерах, hook в error-handler.
- Тесты: `tests/test_audit_log.py` (+ дополнения к существующим, если логируется в них).

## Ограничения / заметки
- Не логировать `BOT_TOKEN`, суммы, имена, личные заметки (приватность). Объём записей малый;
  ротация/архив — отдельная задача при росте. Перед `done`: полный CI, отчёт, лог, push.
