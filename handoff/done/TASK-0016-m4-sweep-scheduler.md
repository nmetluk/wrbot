# TASK-0016: Свип-планировщик уведомлений

## Цель
Связать всё в работающий 24/7 движок уведомлений: периодический свип по БД,
отправка напоминаний с кнопками и идемпотентная фиксация — по ADR-0005.

## Контекст
- ТЗ §1.3, §3.4, §5. Требования: **NFR-1 (24/7), NFR-2 (±1 мин)**, FR-10/FR-11/FR-12.
- ADR-0005 (свип, `sent_reminders`, рестарт-безопасность), ADR-0004 (tz/UTC).
- Готово: due-логика и `SentReminderRepository`/snooze (TASK-0014), текст+кнопки (TASK-0015),
  заглушка `scheduler/app.py`. `__main__` уже стартует `setup_scheduler()`.

## Критерии приёмки (проверяемые)
- [ ] **APScheduler tick:** `AsyncIOScheduler` (tz=UTC), повторяющийся триггер раз в минуту;
      jobstore in-memory допустим — источник истины в БД (ADR-0005), пересоздание при старте.
- [ ] **Свип-корутина:** на каждом тике (UTC-минута) — выбрать пользователей, у кого сейчас
      `notify_time` в их `tz` (helper из TASK-0014) → их активные списания → due `(target_date,
      days_before)` (TASK-0014) → пропустить уже записанные в `sent_reminders` → отправить
      уведомление (текст+кнопки из TASK-0015) → записать в `sent_reminders`. Учесть `snoozed_until==today`.
- [ ] **Идемпотентность/рестарт (NFR-1):** повторный тик/перезапуск процесса в тот же день
      НЕ задваивает напоминания (защита через `sent_reminders`). Покрыто тестом (два прогона свипа → одна отправка).
- [ ] **Изоляция и ошибки:** сбой отправки одному пользователю не роняет весь свип; логируется;
      сессия БД корректно открыта/закрыта (или через middleware-аналог/фабрику).
- [ ] Регистрация свипа в `__main__` (старт/остановка планировщика).
- [ ] **Тесты:** свип выбирает правильных пользователей/списания (mock времени и `bot.send_message`
      через autospec), идемпотентность (2 тика → 1 отправка), snooze-повтор, «нет due → ничего не шлём».
- [ ] Весь CI начисто без BOT_TOKEN зелёный (ruff format/check, mypy, pytest, alembic, validate).

## Ожидаемые артефакты
- Код: `src/wrbot/scheduler/app.py` (триггер), `src/wrbot/scheduler/sweep.py` (или `tasks/sweep.py`),
