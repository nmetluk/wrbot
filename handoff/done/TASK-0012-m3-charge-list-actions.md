# TASK-0012: «Мои списания» и действия над списанием

## Цель
Реализовать «📋 Мои списания» — список ближайших платежей и действия над списанием:
редактировать, удалить, отметить оплаченным (FR-5, FR-6, FR-7).

## Контекст
- ТЗ §3.2 («Мои списания»; действия Редактировать/Удалить/Отметить оплачено;
  «оплачено» → сдвиг даты / закрытие одноразового).
- Требования: FR-5, FR-6, FR-7, FR-13.
- ADR-0006 (сдвиг/мягкое закрытие — уже в `ChargeRepository.mark_paid`, TASK-0010).
- Готово: `ChargeRepository` (TASK-0010), FSM создания (TASK-0011); заглушка `list_charges` в `start.py` (заменить).
- Уроки: специфичные callback-фильтры (`charge_item_<id>`, `charge_paid_<id>`, …) + **тест через роутер**
  (урок TASK-0008); тонкие хэндлеры; тексты в `texts.py`; моки autospec.

## Критерии приёмки (проверяемые)
- [ ] **Список (FR-5):** «📋 Мои списания» показывает активные списания пользователя,
      **отсортированные по `next_date`**; по каждому — название, сумма, кошелёк, категория
      (если есть), следующая дата. Пустой список — понятная подсказка.
- [ ] **Карточка списания** с действиями (FR-6): «✏️ Редактировать», «🗑 Удалить»,
      «✅ Отметить оплаченным».
- [ ] **Оплачено (FR-7):** вызывает `ChargeRepository.mark_paid` — для периодических сдвигает
      `next_date` (с клампом, TASK-0010) и сообщает новую дату; для одноразового — закрывает
      (`status=done`) и убирает из списка. Без двойных списаний (идемпотентность действия).
- [ ] **Редактирование:** изменение полей (название/сумма/кошелёк/категория/дата/период/уведомления)
      — переиспользуя шаги/валидацию из TASK-0011 (через `ChargeRepository.update`).
- [ ] **Удаление:** с подтверждением; изоляция по tg_id (нельзя тронуть чужое).
- [ ] `/cancel`/«Отмена» чистят FSM; все callback-фильтры специфичны (не перехватывают друг друга).
- [ ] Тексты — только `texts.py`; ≥1 тест реального рендера (карточка/список с подстановками).
- [ ] **Тесты:** список (сортировка/пустой), действия (paid периодический и once, edit, delete с
      подтверждением), изоляция tg_id; callback-роутинг — тест через роутер. Зелёный pytest.
- [ ] Весь CI начисто без BOT_TOKEN зелёный (ruff format/check, mypy, pytest, alembic, validate).
