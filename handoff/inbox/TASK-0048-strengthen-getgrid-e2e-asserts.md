# TASK-0048: Усилить позитивные ассерты e2e для /getgrid

## Цель (находка аудита TASK-0047, низкий приоритет, НЕ блокер)
В `tests/test_e2e_smoke.py` (блок TASK-0047) ассертится только сценарий «без упоминания → молчит»
(`len(captured_texts) == 0`). Три позитивных пути — `/getgrid` в группе, @упоминание, `/getgrid` в
привате — **прогоняются через `feed_update`, но НЕ проверяют ответ** (комментарий «response text in
group.py source»). Такой тест зелёный, даже если бот перестанет отвечать или ответит без ID —
ровно класс «слабого теста», который замаскировал баг в TASK-0046.

## Контекст: почему capture не сработал
Перехват через `object.__setattr__(msg, "answer", ...)` не ловит ответ: внутри `feed_update`
aiogram отдаёт хэндлеру копию `Message`, привязанную к боту, и `.answer()` уходит через
`bot(SendMessage(...))`. Надёжный способ захвата (проверено аудитом):
```python
from aiogram.methods import SendMessage
def sent_texts(bot):  # bot = AsyncMock(spec=Bot)
    return [a.text for c in bot.mock_calls for a in c.args if isinstance(a, SendMessage)]
```

## Критерии приёмки (проверяемые)
- [ ] Позитивные сценарии ассертят содержимое ответа:
      - `/getgrid` в supergroup → ответ непустой и содержит строковый ID чата (напр. `-1009876543210`).
      - @упоминание бота в группе → ответ содержит ID чата.
      - `/getgrid` в привате → пришёл hint (непустой), без ID-копипасты группы.
- [ ] Сценарий «без упоминания → молчит» сохранён.
- [ ] Захват ответа надёжный (через `bot(SendMessage)`/mock_calls или эквивалент), а не через
      подменённый `message.answer`, который не вызывается при реальном dispatch.
- [ ] Тест бы падал, если хэндлер перестанет отвечать/уберёт ID (negative-control: убедиться, что
      ассерт чувствителен).
- [ ] Весь CI начисто зелёный.
