# TASK-0017: Свип учитывает notify_time/tz

## Цель
Исправить функциональный блокер: уведомления должны уходить в выбранное пользователем
время `notify_time` в его `tz`, а не на первом тике суток.

## Контекст / приоритет
- **Severity: BLOCKER.** Найдено в AUDIT-M4 (SESSION-2026-06-01-28).
- ТЗ §3.4 («уведомления приходят в выбранное время»), **NFR-2 (±1 мин)**, ADR-0004/0005.

## Суть бага
`run_sweep` → `get_due_reminders_today(session, today, sent_repo)` выбирает due только по
**дате**, для всех активных списаний. Helper `select_users_to_notify_at(session, now_utc)`
(уже реализован в `services/reminders.py`, TASK-0014, и проверяет минутное совпадение
`notify_time` в `tz`) **нигде в свипе не вызывается**. Итог: напоминание уходит на первом
тике суток (≈полночь UTC), а не в `notify_time`.

## Критерии приёмки (проверяемые)
- [ ] Свип отправляет напоминания пользователю **только в ту минуту**, когда его `notify_time`
      в его `tz` совпадает с текущим UTC-моментом (использовать `select_users_to_notify_at`,
      затем считать due только для этих пользователей; либо эквивалентно ограничить
      `get_due_reminders_today` по выбранным пользователям).
- [ ] **Тест тайминга:** при `notify_time=10:00`, `tz=Europe/Moscow` и наступившем due-дне —
      свип НИЧЕГО не шлёт в «не те» минуты (напр. 00:00 UTC) и шлёт ровно в нужную минуту
      (07:00 UTC = 10:00 МСК). Тест должен падать на текущем коде.
- [ ] Несколько поясов: пользователи в разных `tz` получают в своё локальное `notify_time`.
- [ ] Идемпотентность сохранена (повтор тика в ту же минуту/после рестарта → одна отправка).
- [ ] Весь CI начисто без BOT_TOKEN зелёный (ruff format/check, mypy, pytest, alembic, validate).

## Ожидаемые артефакты
- Код: `src/wrbot/scheduler/sweep.py` (+ при необходимости сигнатура `get_due_reminders_today`).
- Тесты: `tests/test_sweep.py` (тайминг + многопоясность).

## Ограничения / заметки
- Не блокировать event loop. Перед `done`: ПОЛНЫЙ прогон CI (`mypy src`, не только новые файлы!), отчёт, лог, push.
