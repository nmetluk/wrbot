# TASK-0043: Схема и UI целей дублирования по категории (ADR-0012)

## Цель (отзыв пользователей, п.4 — часть 1/2)
В настройках категории — возможность добавить/удалить chat_id каналов/групп, куда будут
дублироваться напоминания о списаниях этой категории. (Сама отправка — TASK-0044.)

## Решение (ADR-0012)
- Миграция Alembic: `Category.notify_chat_ids` — nullable JSON/Text (список int chat_id, дефолт пусто).
  Хранить как JSON-строку (как `global_days`/`individual_days` в проекте). Только через Alembic.
- `repositories/categories.py`: методы добавить/удалить/получить chat_id (валидно сериализуя список).
- Настройки категории (handlers/categories.py + клавиатуры): раздел «Каналы для напоминаний» —
  список текущих chat_id с кнопками удаления + кнопка «Добавить» (FSM-ввод chat_id).
- Валидация (bot/validators.py): только **числовой** chat_id (каналы обычно отрицательные, напр.
  `-100…`); `@username` в v1 не поддерживаем — сообщить пользователю. Тексты — через локали.
- Новое FSM-состояние для ввода chat_id (states.py).

## Критерии приёмки (проверяемые)
- [ ] Миграция добавляет `categories.notify_chat_ids` (nullable); `alembic upgrade head` + автоген-диф
      пустой; smoke на Postgres зелёный.
- [ ] В настройках категории видно список целей; можно добавить chat_id (с валидацией) и удалить.
- [ ] Невалидный ввод (не число / @username) → понятная ошибка через локаль, не падает.
- [ ] Доступ к БД — только через репозиторий; список сериализуется/десериализуется корректно
      (пустой = нет целей).
- [ ] Тесты репозитория (add/remove/get, дедуп) + e2e через `dp.feed_update`: добавить цель → видна
      в списке → удалить.
- [ ] Весь CI начисто зелёный.
