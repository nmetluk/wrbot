# handoff/ — конвейер задач между архитектором и исполнителем

Это «почтовое отделение» проекта. Задачи передаются как Markdown-файлы, которые
перемещаются по каталогам-состояниям. Состояние конвейера всегда видно из `git`
и продублировано машиночитаемо в `state/backlog.json`.

## Машина состояний

```
                 /new-task                /take-task                 /end-session (успех)
   (архитектор) ──────────► inbox/ ──────────────► in-progress/ ──────────────────► done/
                                                       │
                                                       │ нужно решение архитектора
                                                       ▼
                                                   blocked/  ──(архитектор отвечает)──► inbox/
```

| Каталог | Кто кладёт | Смысл |
|---------|-----------|-------|
| `inbox/` | архитектор | Новая задача, готова к взятию. Сортировка по имени файла (`TASK-XXXX`). |
| `in-progress/` | исполнитель | Задача взята в работу. Одновременно в работе — обычно одна. |
| `done/` | исполнитель | Выполнено и принято. Рядом — отчёт в `reports/`. |
| `blocked/` | исполнитель | Нужен ответ архитектора (неясность, конфликт с ТЗ, внешняя зависимость). |
| `reports/` | исполнитель | Отчёты `TASK-XXXX-report.md` о выполнении (остаются навсегда). |

## Имя файла задачи
`TASK-XXXX-краткое-латиницей.md`, где `XXXX` — сквозной 4-значный номер (0001, 0002, …).
Номер монотонно растёт и не переиспользуется. Шаблон: `docs/templates/task.template.md`.

## Правила
- Перемещение файла между каталогами делается через `git mv` (история сохраняется).
- После любого перемещения запусти `python scripts/update_state.py` — он пересоберёт
  `state/backlog.json`. CI (`.github/workflows/validate.yml`) проверит соответствие.
- В `done/` задача попадает только при выполненном Definition of Done (см. `CONTRIBUTING.md`).
- Полный протокол: `docs/workflow/handoff-protocol.md`.

## .gitkeep
Пустые каталоги содержат `.gitkeep`, чтобы git их сохранял. Не удаляй эти файлы.
